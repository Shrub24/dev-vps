# Phase 04.1 Research — Beets Inbox-Only Singleton Ingestion

**Date:** 2026-03-31
**Phase:** 04.1-add-beets-inbox-only-singleton-ingestion-phase
**Question:** What implementation shape best satisfies the locked inbox-only Beets worker decisions without changing `/srv/media` authority or introducing promotion flow?

## Summary Recommendation

- Add a dedicated reusable module at `modules/services/beets-inbox.nix` and import it from `modules/applications/music.nix`.
- Run the worker as a dedicated `beets` system user that stays inside existing `music-ingest` / `music-library` boundaries.
- Use `systemd.path` + `oneshot` `systemd.service` for file-event-driven execution on the `slskd` inbox subtree, with a small age/debounce guard inside the runner.
- Keep all mutable Beets state under `/srv/data/beets` via `BEETSDIR=/srv/data/beets`; do not write persistent artifacts into `/srv/media`.
- Keep imports singleton-only and in-place: `write: yes`, `copy: no`, `move: no`, `link: no`, `hardlink: no`.
- Package Beets with a Python environment assembled from nixpkgs packages instead of adding ad-hoc pip installs on the host.

## Locked Decision Fit

- **D-01 / D-04:** Beets importer config must write tags in place and explicitly disable copy/move/link/hardlink behavior.
- **D-02 / D-08:** The worker must run singleton imports only and target `/srv/media/inbox` or narrower inbox subpaths, never `/srv/media` broadly.
- **D-03 / D-12 / D-15:** Low-confidence or skipped files should stay untouched in inbox while timestamped unresolved and summary reports are written under `/srv/data/beets`.
- **D-05 / D-06 / D-07:** Long-term mode should be automatic and file-event driven, while a documented dry-run command remains available for development validation.
- **D-09 / D-10 / D-11:** Discogs, Beatport, and Bandcamp should all be enabled without adding a hardcoded source-priority override.

## Research Findings

### 1. Beets importer can satisfy inbox-only in-place tagging

Beets supports singleton import mode via `beet import -s`. The importer also supports `-C` (do not copy) and default/explicit config for `copy: no`, `move: no`, `link: no`, `hardlink: no`, while still allowing `write: yes`. This is the correct fit for D-01 and D-02 because it updates tags in the existing file path instead of promoting files into a separate managed library path.

Relevant beets config surface:
- `import.write`
- `import.copy`
- `import.move`
- `import.link`
- `import.hardlink`
- `import.quiet`
- `import.quiet_fallback`
- `import.incremental`
- `import.incremental_skip_later`
- CLI `beet import -s`

Planning implication: the module should generate an explicit `config.yaml` with all file-moving knobs disabled rather than relying on defaults.

### 2. Systemd path units are the lowest-complexity automatic trigger that fits repo conventions

The repo already prefers native NixOS services and simple rollout flow over orchestration. A `systemd.path` watching `/srv/media/inbox/slskd` aligns with D-05 and project constraints better than a long-running Python watcher daemon. This keeps the automation declarative, restart-free, and easy to reason about in contract tests.

Recommended shape:
- `systemd.paths.beets-inbox-watch` watches `PathExistsGlob=/srv/media/inbox/slskd/*`
- `systemd.services.beets-inbox-run` is `Type=oneshot`
- service calls a repo-owned runner script that filters to supported audio files and ignores paths younger than a short threshold to reduce partial-download churn

Planning implication: use the path unit for primary automation and expose a separate dry-run invocation path for D-07.

### 3. Discogs is compatible with singleton imports but writes album data onto singletons

Beets Discogs docs explicitly note that Discogs matches are release-based and the album tag is written even to singleton imports. That is acceptable for this phase because the locked decisions require singleton-first matching and conservative merge, not album-field suppression. It does mean the plan should mention this behavior so implementation and contract tests treat it as expected rather than accidental drift.

### 4. Beatport support is built into beets but has an interactive first-time auth step

Beets documents a built-in `beatport` plugin that is useful for electronic music and requires `requests-oauthlib`. The first run after enabling the plugin asks the operator to authorize in a browser and paste a returned line back into the terminal. That is a real user-setup item and should be captured in plan frontmatter rather than hidden.

Planning implication:
- include Beatport auth as `user_setup`
- keep automation structured so the worker can function before Beatport is authorized, but the full desired source set is only complete after the auth step

### 5. Bandcamp support is practical via `beetcamp`, not a built-in beets doc page

The stable beets docs expose Bandcamp external IDs through the MusicBrainz plugin, but not a stable first-party `bandcamp` plugin page. Current NixOS package search shows `python313Packages.beetcamp` / `python314Packages.beetcamp`, and the upstream `beetcamp` README describes it as a Bandcamp autotagger plugin for beets enabled by adding `bandcamp` to the Beets plugin list.

Planning implication:
- Bandcamp support in this phase likely means extending the Beets Python environment with the external `beetcamp` plugin package
- this should be modeled as part of the Nix package composition, not a post-deploy pip install
- because it is an external plugin, plans should include a verification step proving the plugin is available from the generated runtime command surface

### 6. `/srv/data/beets` should own both config and reports through `BEETSDIR`

Beets uses a config/data directory on Unix (`~/.config/beets` by default). Setting `BEETSDIR=/srv/data/beets` gives a clean way to keep config, caches, auth token files, and generated reports inside the approved service-state boundary for D-12 and D-14.

Recommended owned paths:
- `/srv/data/beets/config.yaml`
- `/srv/data/beets/state/` or `/srv/data/beets/cache/`
- `/srv/data/beets/reports/`
- `/srv/data/beets/unresolved/`
- `/srv/data/beets/logs/` only if needed; otherwise rely on journal + report files

Planning implication: use `systemd.tmpfiles.rules` to create `/srv/data/beets` and subdirectories with Beets-owned permissions, and keep report filenames timestamped per D-15.

### 7. Existing repository verification pattern strongly favors shell contract tests plus just aggregation

Phases 03 and 04 already lock behavior using strict-mode shell scripts with `rg --fixed-strings` checks and aggregate them through `just`. This phase should follow the same pattern rather than inventing VM or unit tests.

Planning implication:
- add a dedicated phase-04.1 contract shell script asserting beets module wiring, inbox-only import flags, `/srv/data/beets` state path, and the absence of library-promotion settings
- add a `just verify-phase-04.1` recipe that chains the existing phase-04 checks plus the new beets-specific check

## Recommended Architecture Pattern

### Module boundary

- `modules/services/beets-inbox.nix`
  - owns package/runtime, config file generation, system user/group, tmpfiles, runner script, path/service units
- `modules/applications/music.nix`
  - imports the new service module and grants the Beets service account the exact extra groups needed for inbox writes and library reads
- `hosts/oci-melb-1/default.nix`
  - unchanged except through existing `music.nix` import path

### Runtime model

- Beets runs as a worker, not an always-on daemon
- file arrivals under `slskd` inbox trigger a oneshot import pass
- a separate dry-run service or wrapper command exists for development validation
- unresolved and summary reports are emitted under `/srv/data/beets/reports/<timestamp>-*.json|yaml|txt`

### Package model

Use a Nix-built Python environment, likely one of:
- `pkgs.beets` plus Python packages/plugins in `python3.withPackages`
- or a wrapped script/environment exposing `beet`

Needed package signals from research:
- `pkgs.beets` exists in nixpkgs and supports `aarch64-linux`
- `python3-discogs-client` exists in nixpkgs
- `beetcamp` exists in nixpkgs
- no nixpkgs result was found for a standalone `beatport` package; because built-in beets extras only mention `requests-oauthlib`, the plan should assume the built-in Beets beatport plugin just needs that dependency available in the Python environment

## Common Pitfalls To Avoid

- Do **not** set Beets `directory`/path formats in a way that causes moves out of inbox. This phase is not a library organizer phase.
- Do **not** point imports at `/srv/media` globally. Keep explicit inbox roots only per D-08.
- Do **not** store unresolved markers next to media files. D-12 and D-14 require `/srv/data/beets` only.
- Do **not** make manual `just` recipes the primary steady-state workflow. D-06 says manual runs are only acceptable for seeding/dev.
- Do **not** add SoundCloud now.
- Do **not** add broad ingest/promotion pipeline abstractions; keep this as a narrow worker.
- Do **not** rely on pip-installed mutable state on the host. Keep package composition declarative.

## Validation Architecture

Recommended automated checks for execution plans:
- grep contract assertions that Beets config disables `copy`, `move`, `link`, and `hardlink`
- grep assertions that singleton mode and inbox-targeted roots are encoded in the worker script/service
- grep assertions that `BEETSDIR` and report output paths live under `/srv/data/beets`
- grep assertions that no config/path literal promotes files into `/srv/media` library subpaths outside inbox
- a quick command that renders/greps the generated config or runner script for expected source/plugin names: Discogs, Beatport, Bandcamp
- `just verify-phase-04.1` should include the new contract plus existing `verify-phase-04`
