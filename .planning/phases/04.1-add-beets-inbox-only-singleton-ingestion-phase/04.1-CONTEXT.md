# Phase 04.1: Add Beets inbox-only singleton ingestion phase - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Add an inbox-only Beets ingestion worker for `oci-melb-1` that tags singleton tracks in `/srv/media/inbox` with conservative safety defaults, without introducing library-promotion/move behavior or changing the authoritative media ownership model established in Phase 4.

</domain>

<decisions>
## Implementation Decisions

### Tagging behavior and safety
- **D-01:** Beets must write tags in place under `/srv/media/inbox`; copy/move/link/hardlink behavior is out of scope for this phase.
- **D-02:** Default matching mode is singleton-first.
- **D-03:** Unmatched or low-confidence files remain untouched in inbox and are recorded for manual follow-up.
- **D-04:** Tag replacement policy is conservative merge (avoid blanket overwrite).

### Worker trigger and run cadence
- **D-05:** Target operating model is automatic worker execution triggered by new `slskd` downloads via file watch.
- **D-06:** Manual runs are acceptable for initial seeding/dev iteration, but not the long-term primary mode.
- **D-07:** Dry-run flow is required for development validation, not as a mandatory production gate for every run.
- **D-08:** Import scope stays explicit to inbox targets (`/srv/media/inbox` or selected subpaths), not full-library scans.

### Metadata source policy
- **D-09:** Enable Discogs and Beatport as baseline sources in this phase.
- **D-10:** Candidate selection follows best-score-wins; no hardcoded source priority between Discogs and Beatport.
- **D-11:** Include Bandcamp in this phase; defer SoundCloud.
- **D-12:** Persist unresolved/weak-match outcomes as a structured queue under `/srv/data/beets`.

### Output and audit trail
- **D-13:** Default runtime output is info-level summaries (counts, outcomes, unresolved) rather than per-track debug noise.
- **D-14:** Persistent artifacts live only under `/srv/data/beets`.
- **D-15:** Use timestamped per-run reports and include a concise success change report plus unresolved entries.

### the agent's Discretion
- Exact watch implementation (for example, systemd path unit vs wrapper process) as long as it is file-event driven from inbox/slskd paths.
- Exact report format and filenames under `/srv/data/beets`.
- Exact beets module option names and service wiring details, while preserving the phase decisions above.
- Exact guardrails for retries/backoff/debounce during automatic watch-triggered runs.

</decisions>
