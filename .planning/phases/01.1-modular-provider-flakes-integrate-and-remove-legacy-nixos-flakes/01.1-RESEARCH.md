# Phase 01.1 Research - Modular Provider Flakes + Remove Legacy NixOS Flakes

## Objective

Define the lowest-risk way to modularize provider-specific flake wiring for `oci-melb-1`, absorb the still-relevant `nixos/` artifacts into the active host/module tree, and remove obsolete legacy NixOS files.

## Inputs Reviewed

- `.planning/STATE.md`
- `.planning/ROADMAP.md`
- `.planning/REQUIREMENTS.md`
- `flake.nix`
- `hosts/oci-melb-1/default.nix`
- `modules/core/base.nix`
- `modules/profiles/base-server.nix`
- `modules/services/tailscale.nix`
- `nixos/configuration.nix`
- `nixos/digitalocean.nix`
- `nixos/disko-config.nix`
- `docs/architecture.md`
- `docs/context-history.md`
- `.planning/phases/01-repository-cutover/*-SUMMARY.md`

## Findings

1. **Partial cutover remains:** Active flake/host paths are now fleet-shaped, but `flake.nix` still imports `./nixos/disko-config.nix`, and the legacy `nixos/` tree still exists beside active `hosts/` + `modules/` paths.
2. **Legacy provider module is stale:** `nixos/digitalocean.nix` is no longer imported by `flake.nix` and represents old provider assumptions that conflict with the current first-host OCI target.
3. **`nixos/configuration.nix` is dead-but-risky baggage:** It contains old host identity (`dev-vps`), personal tooling, and legacy service wiring that can confuse future implementation if left in-tree as if active.
4. **Safe migration shape:** Extract remaining reusable/runtime-relevant parts from `nixos/` into active module paths first, then delete legacy files and update docs in the same phase.

## Recommended Implementation Shape

- Introduce explicit provider module boundary under active module tree (for example `modules/providers/oci/default.nix`) and move the disk device/provider defaults there.
- Move disko layout from `nixos/disko-config.nix` to an active module path (for example `modules/storage/disko-root.nix`) and wire that path through host/flake composition.
- Update `flake.nix` so it imports only active host/module paths and no `./nixos/*.nix` files.
- Remove `nixos/configuration.nix`, `nixos/digitalocean.nix`, and `nixos/disko-config.nix` once replacement wiring is in place.
- Update canonical docs (`docs/architecture.md`, `docs/context-history.md`) so source-of-truth paths match code.

## Risks and Mitigations

- **Risk:** Breaking host evaluation during module relocation.
  - **Mitigation:** Keep module move + flake rewiring in one plan with `nix flake check --no-build --no-write-lock-file path:.` per task.
- **Risk:** Accidental deletion of still-referenced legacy files.
  - **Mitigation:** Add explicit grep gate proving zero `./nixos/` imports before deletion completion.
- **Risk:** Documentation drift after structural cleanup.
  - **Mitigation:** Include docs reconciliation task in same phase with exact path checks.

## Validation Architecture

This phase is infrastructure-repo refactoring with command-level verification (no unit test framework):

- Quick loop: `nix flake check --no-build --no-write-lock-file path:.`
- Structural checks:
  - `rg "modules/providers/oci/default.nix|modules/storage/disko-root.nix" flake.nix hosts/oci-melb-1/default.nix`
  - `! rg -q "nixos/disko-config.nix|nixos/digitalocean.nix|nixos/configuration.nix" flake.nix hosts/oci-melb-1/default.nix deploy.sh justfile .github/workflows/ci.yml`
- Full completion check:
  - `nix build --no-link --no-write-lock-file path:.#nixosConfigurations.oci-melb-1.config.system.build.toplevel`

Nyquist implication: every implementation task in this phase should provide one sub-60-second automated verification command and the plan should end with one full host build confirmation.

## Output for Planner

Use two execution plans:

1. **Provider modularization + active wiring** (create provider/storage modules and rewire host+flake)
2. **Legacy NixOS tree retirement + docs reconciliation** (delete obsolete files and align canonical docs)

Each plan should preserve parallel-safe file ownership and include explicit acceptance criteria for removed legacy references.
