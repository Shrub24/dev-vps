# Phase 01.1.1 Research - Legacy Config Migration Cleanup

## Objective

Define the shortest safe cleanup pass that removes legacy configuration leftovers after Phase 01.1 so upcoming secrets/bootstrap work starts from consistent operator and config surfaces.

## Findings

1. **Operator defaults still expose a retired workload name.**
   - `justfile` still defaults `logs` to `unit="codenomad"` even though active baseline services are Tailscale/Syncthing/Navidrome.
2. **Secret scaffolding still uses legacy monolithic naming.**
   - `.sops.yaml` only scopes `secrets/*.yaml`; template file remains `secrets/secrets.template.yaml` and still includes `codenomad`-specific keys.
3. **Planning codebase maps are stale after Phase 01.1 migration.**
   - `.planning/codebase/*` documents still describe deleted `nixos/*.nix` files as active architecture, which degrades planning quality for downstream phases.
4. **Canonical docs are mostly correct, but migration cleanup details are not reflected in planning support artifacts.**
   - `docs/` references active host/provider/storage boundaries, while `.planning/codebase/*` still reflects pre-cutover internals.

## Recommended Implementation Shape

- Normalize operator defaults in `justfile` to active baseline-safe values (no `codenomad` default).
- Migrate secret scaffolding names from legacy `secrets/secrets.template.yaml` toward `secrets/common.template.yaml`, and extend `.sops.yaml` rules for both transitional and host-scoped paths.
- Refresh `.planning/codebase/` architecture maps so future planning agents do not reason from removed `nixos/` paths.
- Keep encrypted payload migration in this phase to naming/scaffolding only; full blast-radius secret split enforcement remains in Phase 2.

## Risks and Mitigations

- **Risk:** Breaking current secret editing workflows while renaming templates/rules.
  - **Mitigation:** Keep transitional `.sops.yaml` compatibility for both legacy and new template path patterns.
- **Risk:** Over-scoping into full Phase 2 secrets implementation.
  - **Mitigation:** Restrict this phase to cleanup and scaffolding; defer host-specific secret material enforcement to Phase 2.
- **Risk:** Docs/plans drift after cleanup.
  - **Mitigation:** Include explicit docs + codebase-map refresh tasks with grep-verifiable acceptance criteria.

## Validation Architecture

This is a config and documentation cleanup phase with static verification:

- Quick loop:
  - `rg "codenomad|tailscale-serve-codenomad" justfile secrets/secrets.template.yaml`
  - `nix flake check --no-build --no-write-lock-file path:.`
- Structural checks:
  - `test -f secrets/common.template.yaml`
  - `rg "secrets/common\.template\.yaml|hosts/<host>/secrets\.yaml|secrets/common\.yaml" docs/architecture.md docs/plan.md`
  - `! rg -q "nixos/configuration.nix|nixos/digitalocean.nix|nixos/disko-config.nix" .planning/codebase/ARCHITECTURE.md .planning/codebase/STRUCTURE.md .planning/codebase/CONCERNS.md`
- Completion check:
  - `nix build --no-link --no-write-lock-file path:.#nixosConfigurations.oci-melb-1.config.system.build.toplevel`

Nyquist implication: each task must include one automated static verification command and the final plan wave must include one host build confirmation.

## Output for Planner

Use two execution plans:

1. **Operator and secret scaffold cleanup** (remove legacy defaults and migrate template/rule naming)
2. **Docs and planning-map reconciliation** (align canonical docs plus `.planning/codebase` architecture artifacts)

Ensure plan tasks keep explicit `<read_first>` and concrete `<acceptance_criteria>` strings for grep-based verification.
