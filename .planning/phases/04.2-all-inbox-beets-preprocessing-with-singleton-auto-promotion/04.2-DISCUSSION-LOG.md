## Post-Execution Decision Update (2026-04-08)

After 04.2 execution, requirements were tightened with explicit acceptance criteria for transfer-lock, settle-delay, and demotion sweep behavior.

Locked update:

- keep canonical promoted destination `/srv/media/library` (lowercase)
- preserve original downloaded filenames during promotion and demotion to avoid playlist breakage
- trigger on inbox modification events
- exit early if any `.tmp` exists under inbox
- apply settle delay before running Beets
- run Beets non-interactive (`-q`) in singleton mode (`-s`)
- sweep leftover inbox audio to `/srv/media/untagged` after Beets run
- enforce single-run concurrency behavior and zero-state inbox for eligible audio after successful runs

## Architecture Simplification Update (2026-04-09)

Implementation direction was further simplified to use native Beets and native systemd semantics wherever possible.

Locked update:

- keep all-inbox targeting (`/srv/media/inbox`) with transfer-safe `.tmp` lockout and settle delay
- use native Beets album import mode (`singletons: no`) with `group_albums: yes`
- tune album matching penalties using Beets `match.distance_weights` (`missing_tracks: 0.1`, `unmatched_tracks: 0.1`)
- use Beets `paths:` for library placement and keep original filenames via Beets template fields
- remove custom runner metadata parsing and custom promotion path construction logic
- rely on native systemd service single-instance behavior instead of custom lockfile/flock logic
- keep post-run sweep of leftover inbox audio to `/srv/media/untagged`
