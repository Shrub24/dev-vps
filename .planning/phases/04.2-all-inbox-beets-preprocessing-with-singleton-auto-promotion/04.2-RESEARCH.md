# Phase 04.2 Research — All-Inbox Beets Preprocessing With Singleton Auto-Promotion

**Date:** 2026-04-01
**Phase:** 04.2-all-inbox-beets-preprocessing-with-singleton-auto-promotion
**Question:** What implementation shape best satisfies the locked 04.2 promotion decisions while preserving the existing `/srv/media` authority model and the phase-04/04.1 contract style?

## Summary Recommendation

- Evolve `modules/services/beets-inbox.nix` instead of introducing a second media-processing service.
- Keep Beets responsible for metadata application and matching, but keep filesystem promotion under the repo-owned runner so filenames stay unchanged per D-08.
- Switch the worker from one batch import pass to **per-file processing** so one hard failure does not block sibling files (D-04).
- Keep `/srv/media` as Navidrome’s `MusicFolder`; promote successful files into `/srv/media/library`, but leave hard failures in place under `/srv/media/inbox` (D-09 through D-13).
- Use `quiet: yes` plus `quiet_fallback: asis` so weak/imperfect metadata can still promote without becoming hard blockers, while real command/runtime errors remain the only failure class that leaves files behind (D-01 through D-03).
- Add a timer backstop alongside the `systemd.path` trigger so all-inbox recursion stays reliable without introducing a heavier watcher daemon.

## Locked Decision Fit

- **D-01 / D-02 / D-03:** A Beets run that exits successfully should still produce promotion, even when metadata is weak. Record fallback/imperfect promotions in reports instead of blocking them.
- **D-04:** Promotion must be per-file. The runner should iterate candidate files one-by-one and keep going after individual failures.
- **D-05 / D-06 / D-07 / D-08:** Promotion destination must be `/srv/media/library/<top-level>/<release>/<original filename>`, with deterministic artist-vs-label heuristics and basename preservation.
- **D-09 / D-10 / D-11 / D-12:** Navidrome should keep `MusicFolder = "/srv/media"`; brief overlap is acceptable and failed inbox items remain playable.
- **D-13 / D-14 / D-15 / D-16:** Hard failures stay in place and are surfaced through `/srv/data/beets` reports. No parking queue, no review UI, no manual reset state.

## Research Findings

### 1. Beets already supports the quiet singleton import posture needed for non-blocking promotion

The beets tagger guide confirms:

- `beet import -s` runs singleton mode.
- `beet import -C` leaves files where they are.
- `beet import -q` runs unattended and relies on `import.quiet_fallback` for ambiguous cases.

This means 04.2 does **not** need a custom matcher. It can keep Beets for matching/tag writing, then run a separate repo-owned move step only after Beets succeeds.

Planning implication:

- Keep Beets import as `beet -c /srv/data/beets/config.yaml import -q -s -C`.
- Set `import.quiet_fallback: asis` so ambiguous-but-usable files still succeed and can promote.
- Define “hard failure” narrowly as non-zero import exit, unreadable metadata, destination-creation failure, or move failure.

### 2. Filename preservation is easier in the wrapper than in Beets path formats

Beets path formats are strong for directory design, but they generate the full destination path—including filename. The path format reference documents metadata-derived path templates and auto-appended extensions, but it does not provide a simple built-in “preserve original basename while still reorganizing only directories” contract.

Planning implication:

- Keep `move: no` inside Beets.
- After Beets finishes for one file, compute the destination directory in the wrapper and `mv` the original file there while preserving `basename "$file"`.
- This directly satisfies D-08 without depending on undocumented path-template behavior.

### 3. A metadata-to-path helper can stay deterministic and simple

The user explicitly left the exact artist-vs-label rule to agent discretion. A narrow deterministic heuristic is enough:

- Prefer `label` as the top-level directory when `label` is present and either `albumartist` is missing, `albumartist` is `Various Artists`, or `albumartist` looks compilation-like / weak for catalog navigation.
- Otherwise use `albumartist`, falling back to `artist`.
- Use `album` as the release directory when present.
- If `album` is missing, fall back to `title` or the source stem.

Planning implication:

- Implement the placement calculation in a small embedded Python helper using Beets/mediafile metadata already available in the runtime.
- Output destination: `/srv/media/library/<chosen-top-level>/<chosen-release>/<original-basename>`.

### 4. The `fromfilename` plugin is useful now that weak metadata should still promote

The fromfilename plugin exists specifically to infer artist/title/track info from filenames when tags are poor or absent. That lines up with D-01, D-02, and D-07 because it increases the chance of “usable enough” promotion without forcing manual review.

Planning implication:

- Add `fromfilename` to the active Beets plugin list for 04.2.
- Keep Discogs, Beatport, and Bandcamp in place.
- Treat filename-derived placement as normal fallback behavior, not as a hard-failure case.

### 5. All-inbox automation needs a backstop because `systemd.path` globbing alone is shallow

The current phase-04.1 worker watches `/srv/media/inbox/slskd/*`. For 04.2 the scope expands to all of `/srv/media/inbox`, including nested producer folders. A single top-level path glob is not enough to guarantee recursive change capture for every nested write pattern.

Planning implication:

- Keep a path-triggered fast path for responsiveness.
- Add a low-frequency timer backstop (for example every 15 minutes) that re-runs the same oneshot worker against `/srv/media/inbox`.
- This stays within the project’s “native systemd first” posture and avoids a long-running watcher daemon.

### 6. Existing verification patterns still fit: shell contract tests + just aggregation

Phase 03, 04, and 04.1 all enforce behavior with strict-mode bash scripts and `just` aggregation. 04.2 should preserve that pattern instead of adding VM or unit-test machinery.

Planning implication:

- Add a new `tests/phase-04.2-beets-promotion-contract.sh` for promotion-specific assertions.
- Update service-flow drift coverage so Navidrome remains `/srv/media`-wide while `/srv/media/library` becomes the intended promoted subtree.
- Add `just verify-phase-04-2` with the same dotted-name workaround pattern used in 04.1.

## Recommended Architecture Pattern

### Runtime shape

- `modules/services/beets-inbox.nix`
  - still owns package assembly, generated config, runner script, systemd units, and `/srv/data/beets` state.
  - now also owns `/srv/media/library` bootstrap plus promotion/report logic.
- `modules/applications/music.nix`
  - remains the application composition boundary.
  - continues to own generic inbox identity and group membership.
- `modules/services/navidrome.nix`
  - stays on `MusicFolder = "/srv/media"`.

### Runner model

- scan `/srv/media/inbox` recursively for supported audio files older than 120 seconds.
- process each file independently:
  1. run Beets singleton import in quiet mode against that file.
  2. if import succeeds, compute canonical library destination.
  3. move the same original file into `/srv/media/library/.../<original basename>`.
  4. record whether promotion used strong metadata vs fallback metadata.
  5. if any hard error occurs, leave file in place and append a failure record.

### Reporting model

- Keep all reports under `/srv/data/beets`.
- Recommended concrete files per run:
  - `/srv/data/beets/reports/<timestamp>-summary.txt`
  - `/srv/data/beets/reports/<timestamp>-promoted-with-fallback.tsv`
  - `/srv/data/beets/unresolved/<timestamp>-hard-failures.tsv`

## Common Pitfalls To Avoid

- Do **not** narrow Navidrome to `/srv/media/library`; D-09 explicitly keeps playback visibility on all of `/srv/media`.
- Do **not** use Beets `move: yes` if the resulting path logic would rename files.
- Do **not** reintroduce a special manual-review queue, parked state, or retry suppression mechanism.
- Do **not** preserve `slskd` or other source-folder identity in final library paths.
- Do **not** treat metadata weakness alone as a blocker.
- Do **not** replace native systemd triggers with a bespoke daemon.

## Validation Architecture

Recommended automated checks for execution plans:

- a shell contract asserting all-inbox scope, timer/path backstop, and per-file continuation behavior.
- grep checks that promotion root is `/srv/media/library`, failed files remain rooted in inbox, and filenames are preserved by basename-based move logic.
- grep checks that `quiet_fallback: asis`, `plugins: discogs beatport bandcamp fromfilename`, and `/srv/data/beets` reporting roots are present.
- service-flow checks that Navidrome still reads `/srv/media`, not `/srv/media/library` only.
- `just verify-phase-04-2` should aggregate `just verify-phase-04` plus the new 04.2 promotion contract.
